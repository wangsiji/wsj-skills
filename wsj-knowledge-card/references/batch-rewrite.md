## 批量重写已有卡片

当用户要求**将一批已有卡片升级到新格式**时（如补充视觉化表格、重新排版、调整 callout 类型），使用以下流程：

### 流程

```
收到请求（如"重写 8 张卡片为新格式"）
  ↓
Step 1: 搜索所有源卡片文件
  search_files(path="~/projects/wsj-second-brain/03-Resources/Evergreen",
               pattern="{卡片名}")
  ↓
Step 2: 逐张读取源卡片
  read_file(path="{卡路径}")
  ↓
Step 3: 对每张卡片，将旧内容映射到新格式 7 节
  保留：核心概念定义、来源依据、个人体验、行动指引、参考资料
  重构：将旧条目归类到新模板的对应节
  新增：Problem 警示表、来源表格、误解表格、关系表格、边界表、留空 Insight
  ↓
Step 4: 输出全部卡片到同一文件（或逐文件覆盖）
  - 用 `### 卡片：名称` 作为卡片分隔
  - **卡片之间不用 `---` 分割线**
  - 每张卡片独立 frontmatter（简化版：无 source_type/confidence/created/modified）
  ↓
Step 5: 验证
  - `grep -c "^### 卡片："` = 卡片总数
  - `grep -c "^---$"` = 卡片数 × 2（仅 frontmatter 分隔符，无多余分割线）
  - `grep -c "⚠️ 此部分由你填写"` = 卡片总数（每卡均有留空 Insight）
```

### 格式映射

| 旧卡片内容 | 新格式目标节 | 说明 |
|------------|------------|------|
| 核心概念下的要点 | Concept（📖 定义 / 🔍 本质） | 单点展开为段落 |
| 个人体验 | Example（`[!example]`） | 转化为第三人称案例，不自称"你" |
| 行动指引 | Action（`[!check]` + 步骤表） | 转化为步骤表格 |
| 参考资料 | Concept（🔬 来源与依据） | 转化为来源表格 |
| —（新增） | Problem（`[!warning]` + 以为/实际表） | 从核心概念推导常见误解 |
| —（新增） | Relation（类型表） | 扫描已有 vault 建立 [[wikilinks]] |
| —（新增） | Boundary（✅/❌ 表） | 判断适用/不适用场景 |
| —（留空） | Insight（`[!abstract]`） | 保留模板，不填充 |

### 常见陷阱

- ❌ 在卡片之间加 `---` 视觉分割线 → 使用 `### 卡片：` 标题自然分隔
- ❌ 在 Insight 节填充内容 → 必须留为空白模板
- ❌ **批量重写全部 Insight 被 AI 污染** — 2026-07-10 发现 50 张卡 Insight 全部被填了 AI 生成的问题（「我在哪个场景…」「我最近一次…」）。根本原因：子 agent 批量重写时 Insight 留空规则未被强制执行。修复：全量清洗为统一空模板。**批量重写后必须逐张验证 Insight 尾部无 AI 内容。**
  - 验证命令（在 `03-Resources/Evergreen/` 目录执行）：
    ```bash
    for f in *.md; do
      after=$(sed -n '/^## 7. 内化/,$p' "$f" | tail -n +2)
      real=$(echo "$after" | grep -v '⚠️\|你填写\|AI 提炼\|维度\|理解\|经历\|迁移\|^|-')
      [ -n "$(echo "$real" | grep -v '^$')" ] && echo "⚠️ $f"
    done
    ```
    输出为空 = 全部干净。有输出 = 对应卡片还有 AI 内容，逐个清洗。
- ❌ **不要删除原始任务项** — 马拉松/旅居/王轻舟等日志笔记中的 `- [x]` 和 `- [ ]` 任务项是核心数据，只能重组格式不能删除。笔记整理模式：顶部仪表盘总览 + 原始数据用 `[!info]-` 折叠保留。
- ❌ 保留旧 frontmatter 的 `created`/`modified`/`source_type`/`confidence` → 仅保留简化版字段
- ❌ 忽略 vault 中已有的 [[双链]] 关系 → 必须用 `search_files` 扫描 existing vault
- ❌ 多张卡片输出时混在一起 → 每张卡片起始行必须是 `### 卡片：{名称}`
- ❌ **编号级联错误** — 写完 Concept（节 2）后可能误标 Relation 为「4」而非「3」，造成后续连续错位。写完后 grep 验证：`grep "^### [1-7]\."` 应正好 7 行，按序 1→2→3→4→5→6→7。
- ❌ **遗漏「7. 内化（Insight）」标题** — `> [!abstract]` 上方容易忘记加 `### 7. 内化（Insight）` 标题头。卡片可能通过了「Insight 留空检查」（因为有 `⚠️`），但实际缺少标题头。额外验证：`grep "### 7\. 内化"` 应为卡片总数。
- ❌ **旧 format 残留** — 旧卡片可能使用 `- 核心概念` / `- 个人体验` / `- 行动指引` / `- 参考资料` 无序列表结构，以及 frontmatter 中 `description: dv.view(...)`、旧 tags（如 `[百度工作经验, 百度算法工程师]`）。迁移后需清理：description 替换为真实句子、tags 统一为 `[常青卡片]`。
- ❌ **区块标题层级错误** — Subagent 可能误用 H3 而非 H2，导致整张卡缺少主节标题。这是最常见的批处理翻车场景。
- ❌ **重复节标题** — 后处理脚本（如 sed/python regex）可能误插重复的节标题，如 `## 6. 边界` 出现两次。验证：`grep -c "^## 6\\."` = 1，对所有 1-7 节逐一确认。
